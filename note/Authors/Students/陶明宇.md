在之前的基础上加以补充

1，程序文件的更新
	和个人的PC端的NOTE相仿，高能所的远程服务器上的文件也是和git库的文件相捆绑的，所以在正式跑文件的时候，应该检查有关文件是不是最新版本。
	就是利用和更新本地从远程git库的文件的方法一样。
	git remote update 
	开始下载远程的git库里面的文件
	之后可能要进行身份验证，输入自己高能所的和GitHub上的账号密码（一共输入两个的）开始更新。
	git merge raser/main
	将自己在git库里面克隆的文件名  输入git merge 后面，此处高能所组的文件名是raser/main ，所输入的文件名称不是你命名的那个文件夹的名称。

2，程序的运行位置
	之前说的是具体的操作，没说明一些其他的细节，
	实际上我们的文件运行的位置也是有要求的，我们打开的初始文件位是/afs/ihep.ac.cn/users/（你的名字）
	但实际上我们跑程序的位置是在/scratchfs/atlas/(你的名字)这个地方。
	可以理解为C盘和D盘间的关系。
	如果在/afs/ihep.ac.cn/users/这个下面跑，那么可能因为文件过大导致爆盘，运行失败。

3，在/scratchfs/atlas/(名字）里面运行程序的前瞻性配置
	其实和/afs/ihep.ac.cn/users/中clone GitHub中的文件一样，在此我们也需要建立一个克隆，以后直接在此跑程序，具体的方法参考Note-Knowledge Base-Computer_李星臣部分的GitHub Clone介绍

7.前台和后台的区别

前台：不能退出，直接运行文件，必须等他跑完，跑出来的结果在raser/fig里面

后台：可以退出，在哪里挂着，排队跑，跑出来的结果在raser/output里面

两者的区别在于运行的文件里面有没有for循环，有就是跑作业没有就是前台运行程序

8,关于查询后台作业运行情况

在交完程序上传完之后，用run（103行）的export PATH=/afs/ihep.ac.cn/soft/common/sysgroup/hep_job/bin:$PATH 和 hep_q - u 查询自身作业情况（q 和u中间有个空格)

利用hep_q可以查询总的作业排队情况

而有时会出现不同的错误，如下图，就是昨夜被挂了的情况
![](https://raser-1314796952.cos.ap-beijing.myqcloud.com/media/error%20in%20held.png)
可以用hep_q -u held 命令查询作业被held的原因

hep_rm -a 可以删除后台运行的文件

**（**定位  Beam Conditions Monitor for ATLAS  5.1.1**）**

### 一，**前瞻性补充说明**

1，以下的解释使用24 GeV/c 的中子作为辐照源

2，换算成某种新的单位是Φeq [1015 cm−2 ] 其代表含义是每cm2中1015个能量为1GeV的中子的辐照损伤程度

3，使用的器件和电路图

器件：

1，Hewlett Packard LCR meter (HP 4284A)

作用：一个LCR电路的控制元件，质量挺好的，反正就是控制频率之类的电子线路物理量

2，Keithley 237 source-measure unit：

作用一个元件作用：提供源，很准确的那种

3，一个受到辐照的探测器

4，不同频率和幅值的电压

1，将半导体看做一个有完全耗尽和非完全耗尽的半导体串联，、

我们可以认为耗尽的半导体区域为电容，非耗尽的为电阻

右侧的电容电阻和耗尽电压与外加电压的关系为

(Ru∝(1 − pV /VF D), Cu∝1/(1 − pV /VF D)),

2，通过控制外加电压的频率和幅值来控制耗尽区和非耗尽区的程度，电压越大耗尽区越大

但电压给的是耗尽区和非耗尽区串联的总电压，不能单独测量耗尽区和非耗尽区的，而在之后的实验结果分析中也将说到这个
